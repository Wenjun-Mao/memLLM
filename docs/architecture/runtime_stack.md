# Runtime Stack

## Purpose

This document explains what the phase-1 stack is doing at runtime, how Letta memory affects the
final reply call, what the development UI is showing, and which knobs currently control payload
size.

## Stack Roles

- `Streamlit dev UI`: operator-facing chat and debug surface.
- `FastAPI API`: orchestration layer for character resolution, session resolution, retrieval,
  reply generation, and memory persistence.
- `Letta`: long-term memory layer. It stores shared character memory plus pair-specific user
  memory and retrieval passages.
- `Ollama`: local model runtime. In phase 1 it is used for Letta-side memory work and the local
  memory extractor.
- `External reply provider`: optional user-facing reply model. For example, `winter_poet` and
  `lin_xiaotang` send the final reply request to the DouBao endpoint.
- `Postgres/pgvector`: Docker-hosted storage for Letta and app metadata.

## One Chat Round

1. The UI sends `user_id`, `character_id`, and `message` to the API.
2. The API resolves the character manifest from app metadata.
3. The API resolves or creates the Letta agent for that exact `(user_id, character_id)` pair.
4. Letta returns the current memory context for the turn:
   - shared blocks such as `persona`, `style`, `role`, `background`, `lore`
   - the pair-specific `human` block
   - top-k retrieved archival passages relevant to the new user message
5. The API builds the final reply-provider request.
6. The final user-facing reply is generated by the configured provider:
   - local Ollama for `ollama_chat`
   - the external HTTP endpoint for `custom_simple_http`
7. The reply is returned to the UI immediately.
8. After that, the local memory extractor produces a `MemoryDelta` and writes it back to Letta.
9. The app metadata store records the finished turn.

## What Letta Stores

### Shared character memory

This is seeded from each character manifest and reused across every user talking to that
character.

- `persona`
- optional style or role blocks such as `style`, `role`, `background`
- synthesized `lore` from `shared_passages`

### Pair-specific memory

This belongs to exactly one `(user_id, character_id)` pair.

- `human` block: the current durable summary of the user relationship and preferences
- archival passages: stored snippets from earlier turns that can be retrieved later
- app session record: the mapping from `(user_id, character_id)` to the Letta `agent_id`
- app chat-turn rows: the turn log used for recent message history

## How Memory Reaches the Final Call

The final provider call is not sent raw Letta objects. The API flattens the current memory context
into a prompt-friendly shape.

For the external simple HTTP provider, the final request is built as:

- `system_content`: character persona plus system prompt and a consistency instruction
- `user_content`: character name, memory blocks, retrieved passages, and recent conversation

That is why the `Final Provider Call` panel in the dev UI is useful: it shows the exact last
payload that left the app after memory retrieval and request assembly.

## What the Dev UI Panels Mean

- `Chat`: only the user-visible conversation.
- `Character`: the repo-defined manifest config, not live memory.
- `User-Agent Pair`: the Letta agent currently backing one user talking to one character.
- `Final Provider Call`: the exact last outbound request to DouBao or local Ollama for that turn.
- `Middle Steps`: the temporary per-turn orchestration trace.
- `Memory Snapshot`: the current Letta-backed memory view for the selected pair.

## How Retrieval Works

At reply time, the API asks Letta for:

- all current memory blocks attached to the pair's agent
- top-k relevant passages for the current user message

Those retrieved blocks and passages are then included in the final provider payload. This means
retrieval quality depends both on what is stored in Letta and on how much of it the API includes.

## Length Controls

The current stack has some practical size controls, but not a full token-budgeting layer yet.

### Controls that already exist

- Shared block length:
  - edit the actual text in the character manifest, such as `persona`, `system_prompt`,
    `shared_blocks`, and `shared_passages`
  - these values are currently inserted more or less verbatim into the prompt-building path
- Initial user memory length:
  - `memory.initial_human_block` in each character manifest controls the starting `human` block
- Retrieved passage count:
  - `memory.archival_search_limit` controls how many passages Letta retrieves for a live turn
- Memory snapshot size in the UI:
  - `memory.snapshot_passage_limit` controls how many passages are returned in the snapshot view
- Recent conversation size:
  - `memory.recent_message_window` controls how many recent turns are included in the final reply
    request
- Letta local-model generation size:
  - `MEMLLM_API_LETTA_MODEL_MAX_TOKENS` controls the Letta-side model generation ceiling
  - `MEMLLM_API_LETTA_MODEL_CONTEXT_WINDOW` controls the configured Letta-side context window
- Local Ollama final-reply generation size:
  - add provider options such as `num_predict` under `reply_provider.extra` for an
    `ollama_chat` character
- External provider-specific generation size:
  - `custom_simple_http` forwards `reply_provider.extra` as-is to the endpoint
  - if the external API supports its own length parameter, it can be placed there

### Controls that do not exist yet

These are the main current gaps.

- No hard token or character cap per memory block before prompt assembly
- No hard token or character cap per retrieved passage before prompt assembly
- No single total-budget pass that trims the final provider payload to a target token size
- No hard cap on memory extractor output size beyond prompt guidance and model behavior

### Practical rule of thumb today

If the final provider call is getting too large, the current tuning order is:

1. shorten `persona`, `system_prompt`, and other shared blocks in the manifest
2. reduce `memory.archival_search_limit`
3. reduce `memory.recent_message_window`
4. shorten `memory.initial_human_block`
5. add provider-specific output caps such as `num_predict` for local Ollama

### Recommended next step if we need strict budgets

The next real upgrade would be a dedicated truncation layer before the final provider call is sent.
That layer should apply explicit limits to:

- each block value
- each passage text
- recent conversation history
- the total assembled payload

That work would belong in the prompt-building path, not in the UI.
